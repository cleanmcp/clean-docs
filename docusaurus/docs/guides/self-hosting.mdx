## Docker deployment

The Docker image is the recommended way to run Clean in production. It pre-caches the embedding model so startup is fast.

### Build and run

```bash
git clone https://github.com/useclean/Clean.git
cd Clean

cp .env.example .env
# Edit .env — set CLEAN_API_KEY, CLEAN_LICENSE_KEY, and CLEAN_ORG_TOKEN
```

```bash
docker compose up -d
```

The API is available at `http://localhost:8000`.

### What's in the image

The Dockerfile uses a multi-stage build:

1. **Builder stage** — installs dependencies, downloads the embedding model (~90MB), compiles Python to bytecode
2. **Runtime stage** — minimal `python:3.12-slim` with only git and compiled packages

The embedding model (`all-MiniLM-L6-v2`) is pre-downloaded at build time, so the container starts instantly without network calls.

### Data persistence

Data is stored in a Docker volume mounted at `/data/clean/`:

| Path | Contents |
|------|----------|
| `/data/clean/index/` | LanceDB vector tables |
| `/data/clean/repos/` | Cloned repositories |
| `/data/clean/metadata.db` | SQLite (projects, jobs, keys) |

The `docker-compose.yml` creates a named volume `clean-data` for persistence:

```yaml
services:
  clean:
    build: .
    ports:
      - "8000:8000"
    volumes:
      - clean-data:/data/clean
    environment:
      - CLEAN_LICENSE_KEY=${CLEAN_LICENSE_KEY}
    env_file:
      - path: .env
        required: false
    restart: unless-stopped

volumes:
  clean-data:
```

### Health checks

The container exposes two health endpoints:

| Endpoint | Auth | Checks |
|----------|------|--------|
| `GET /` | No | Basic liveness (returns `{"status": "ok"}`) |
| `GET /health` | No | Deep check — verifies embedding model loaded and LanceDB accessible |

Use `GET /health` for Docker health checks or load balancer probes:

```bash
curl http://localhost:8000/health
```

Returns `200` when healthy, `503` when degraded.

### Development mode

For development, build with `CLEAN_BUILD_MODE=dev` to skip license and auth requirements:

```bash
docker build --build-arg CLEAN_BUILD_MODE=dev -t clean:dev .
docker run -p 8000:8000 clean:dev
```

Or use `make dev` to run directly with Python (automatically uses dev mode).

## Direct Python deployment

If you prefer not to use Docker:

```bash
pip install "clean[api]"
```

Set environment variables and start the server:

```bash
export CLEAN_API_KEY=your-secure-key
export CLEAN_LICENSE_KEY=your-license-key
export CLEAN_ORG_TOKEN=your-org-token

clean-api
```

The server starts on `0.0.0.0:8000` by default. Override with `CLEAN_API_HOST` and `CLEAN_API_PORT`.

### Resource requirements

| Resource | Minimum | Recommended |
|----------|---------|-------------|
| RAM | 2GB | 4GB+ |
| CPU | 2 cores | 4+ cores |
| Disk | 1GB + repo sizes | SSD recommended |

The embedding model uses ~500MB of RAM. Each indexed repository requires disk space for the cloned repo plus the vector index (~10-50MB depending on codebase size).

### Reverse proxy

If running behind nginx or similar:

```nginx
location / {
    proxy_pass http://localhost:8000;
    proxy_set_header Host $host;
    proxy_set_header X-Real-IP $remote_addr;
}

# SSE requires special handling
location /mcp/sse {
    proxy_pass http://localhost:8000;
    proxy_set_header Connection '';
    proxy_http_version 1.1;
    chunked_transfer_encoding off;
    proxy_buffering off;
    proxy_cache off;
}
```

:::warning
MCP over SSE requires that your reverse proxy does **not** buffer the response. Set `proxy_buffering off` and disable chunked transfer encoding for the `/mcp/sse` endpoint.
:::

## Production requirements

When running Clean in production (i.e. not a dev build), the following apply:

### Gateway connection

A valid `CLEAN_ORG_TOKEN` is required. On startup, Clean authenticates with the gateway within **60 seconds** — if the connection cannot be established, the server will not start. If the gateway connection drops after boot, Clean continues operating with a **1-hour grace period** before shutting down.

### License validation

Clean performs periodic license validation via a phone-home check:

- **Frequency**: Every hour
- **Offline tolerance**: 24 hours — if the license server is unreachable for more than 24 hours, search functionality is disabled until connectivity is restored
- **What's sent**: License key hash, org token, instance ID, and current repo count (no code or query data is transmitted)
