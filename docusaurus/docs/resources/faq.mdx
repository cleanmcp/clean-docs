<details>
<summary><strong>What languages does Clean support?</strong></summary>

Clean currently supports **Python** (`.py`), **JavaScript** (`.js`, `.jsx`, `.mjs`, `.cjs`), and **TypeScript** (`.ts`, `.tsx`). Support for Go, Rust, and Java is on the [roadmap](/resources/roadmap).

</details>

<details>
<summary><strong>Does my code leave my machine?</strong></summary>

No. Clean runs entirely on your infrastructure. The embedding model (`all-MiniLM-L6-v2`) runs locally on CPU — no API calls to external services. Your code is cloned to your local disk and indexed into a local LanceDB database.

</details>

<details>
<summary><strong>How large of a codebase can Clean handle?</strong></summary>

Clean has been tested with repositories containing 10,000+ files. The embedding model processes entities in batches of 5,000. Indexing time depends on your hardware — a 5,000-file repo typically indexes in 1-3 minutes.

</details>

<details>
<summary><strong>How does Clean differ from grep or code search?</strong></summary>

Clean searches by **meaning**, not text. A query like "function that handles user authentication" will find `login()`, `verify_credentials()`, and `authenticate_request()` even though none of those names contain "authentication". Plus, Clean returns the call graph — who calls the function and what it calls — so your agent understands the full context.

</details>

<details>
<summary><strong>What is TOON format?</strong></summary>

TOON (Token-Optimized Object Notation) is Clean's compact output format. It uses a tabular layout instead of JSON, saving 30-40% of tokens on every search response. This matters because every token your AI agent reads costs money. See [TOON format](/concepts/toon-format) for details.

</details>

<details>
<summary><strong>Do I need a license key?</strong></summary>

For production use, yes. Set `CLEAN_LICENSE_KEY` in your environment. For development, use a dev build (`make dev` or `docker build --build-arg CLEAN_BUILD_MODE=dev`) which skips license validation.

</details>

<details>
<summary><strong>How does incremental indexing work?</strong></summary>

When you search, Clean checks if the index is stale by comparing the current git HEAD against the stored commit SHA. If the code has changed, it uses `git diff` to identify modified files and only re-parses and re-embeds those files. This is typically much faster than a full re-index.

</details>

<details>
<summary><strong>Can I index private repositories?</strong></summary>

Yes, through the GitHub App integration. Configure a GitHub App with read access to your repositories, and Clean will use installation tokens to clone private repos. See [GitHub integration](/guides/github-integration).

</details>

<details>
<summary><strong>What's the difference between SSE and stdio transport?</strong></summary>

**SSE** (Server-Sent Events) runs Clean as an HTTP server that agents connect to remotely. Best for multi-user setups or when the agent and Clean run on different machines.

**Stdio** runs the `clean` CLI command directly — your agent spawns the process and communicates over stdin/stdout. Best for local single-user setups with no HTTP overhead.

</details>

<details>
<summary><strong>How do I reset the index for a repository?</strong></summary>

Delete and re-index:

```bash
# Delete
curl -X DELETE "http://localhost:8000/repos/owner/repo" \
  -H "Authorization: Bearer YOUR_API_KEY"

# Re-index
curl -X POST "http://localhost:8000/repos/index?repo=owner/repo" \
  -H "Authorization: Bearer YOUR_API_KEY"
```

</details>
