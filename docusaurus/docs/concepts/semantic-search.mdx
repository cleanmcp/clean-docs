## How search works

When you (or your AI agent) searches Clean, three things happen:

1. **Embed the query** — your natural language query is converted to a 384-dimensional vector using the same model that embedded your code (`all-MiniLM-L6-v2`).

2. **Vector search** — LanceDB finds the top-K code entities closest to your query vector using L2 (Euclidean) distance, converted to a similarity score: `similarity = 1 / (1 + distance)`.

3. **Context expansion** — if `depth > 0`, Clean follows the call graph around the best result to include callers, callees, and same-file entities.

## Search parameters

| Parameter | Default | Range | Description |
|-----------|---------|-------|-------------|
| `q` / `query` | — | 1-1000 chars | Natural language search query |
| `repo` | — | `owner/repo` | Repository to search |
| `top_k` | 5 | 1-50 | Number of results to return |
| `depth` | 1 | 0-5 | Call graph expansion depth |
| `branch` | default | — | Git branch (if multiple indexed) |

## Context expansion

This is Clean's key differentiator. Instead of just returning matching functions, Clean shows you the **call neighborhood** — what a function calls, what calls it, and what else lives in the same file.

```
                      callers
                    ┌──────────┐
                    │ route()  │  ← who calls login()?
                    └────┬─────┘
                         │
                         ▼
  same file         ┌──────────┐         callees
┌───────────┐       │ login()  │       ┌──────────────┐
│ logout()  │  ←──  │  (match) │  ──►  │ validate()   │
│ refresh() │       └──────────┘       │ create_session│
└───────────┘                          └──────────────┘
```

At `depth=1`, you get the immediate callers and callees. At `depth=2`, you also get *their* callers and callees, and so on up to `depth=5`. The maximum is capped at 50 entities per direction to keep responses fast.

**Example response with context:**

```
results
  name | file | line | sim
  login | src/auth/handler.py | 42 | 95%

context
  function: login() in src/auth/handler.py:42-67

  callees:
    validate_credentials | src/auth/validator.py | 12
    create_session | src/auth/session.py | 28
    log_login_attempt | src/audit/logger.py | 55

  callers:
    handle_login_route | src/routes/auth.py | 15

  same_file:
    logout | src/auth/handler.py | 70
    refresh_token | src/auth/handler.py | 95
```

## Similarity scoring

Clean uses L2 distance internally, converted to a 0-1 similarity score:

```
similarity = 1.0 / (1.0 + l2_distance)
```

- **0.90+**: Strong semantic match
- **0.70-0.90**: Related code
- **< 0.70**: Weak match

Results are sorted by similarity descending.

## Staleness detection

Before returning results, Clean checks if the index is stale:

1. Compare the current git `HEAD` against the stored commit SHA
2. Check for uncommitted changes via `git status --porcelain`
3. If stale, trigger an incremental re-index before searching

This means agents always search against up-to-date code, without manual re-indexing.
