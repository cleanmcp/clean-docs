## Architecture

Clean is a semantic code search engine built for AI agents. It has three layers:

```
┌─────────────────────────────────────────────────┐
│  TRANSPORT       FastAPI (HTTP) + MCP (stdio/SSE) │
├─────────────────────────────────────────────────┤
│  SEARCH          Vector search + context expansion │
├─────────────────────────────────────────────────┤
│  INDEX           Parse → Embed → Store → Call graph │
├─────────────────────────────────────────────────┤
│  STORAGE         LanceDB (vectors) + SQLite (metadata) │
└─────────────────────────────────────────────────┘
```

**Transport** exposes two interfaces: a FastAPI HTTP server (REST endpoints + MCP over SSE) and a standalone MCP server (stdio for direct agent integration).

**Search** takes a natural language query, embeds it, runs vector search against LanceDB, and optionally expands results with call graph context.

**Index** orchestrates the five-phase pipeline that turns source code into searchable vectors.

**Storage** uses LanceDB for vector data and SQLite for project metadata, job state, and API keys.

## Indexing pipeline

When you index a repository, Clean runs five phases:

<Steps>
  <Step title="Scan">
    Walk the directory tree. Filter by supported extensions (`.py`, `.js`, `.jsx`, `.mjs`, `.cjs`, `.ts`, `.tsx`). Skip `node_modules`, `.git`, `__pycache__`, `dist`, `build`, and 40+ other patterns. Validate file safety (size under 10MB, no symlink escapes, no sensitive files like `.env` or `.pem`).
  </Step>
  <Step title="Parse">
    Feed each file into the appropriate **tree-sitter** parser. Extract `CodeEntity` objects: functions, methods, classes, interfaces, type aliases, arrow functions. For each entity, record the name, file path, line range, source code, and a list of function calls it makes.
  </Step>
  <Step title="Embed">
    Batch entities (up to 5,000 per batch) and embed them using **all-MiniLM-L6-v2** — a 384-dimensional sentence-transformer that runs locally on CPU. No API calls, no data leaves your machine.
  </Step>
  <Step title="Store">
    Upsert entities into **LanceDB** via merge-insert (deduplicates by entity ID). Each entity is stored with its embedding vector, metadata, and call list.
  </Step>
  <Step title="Compute call graph">
    Build reverse mappings: for each entity, compute `called_by` from the `calls` lists of all other entities. This enables bidirectional context expansion at search time.
  </Step>
</Steps>

## Data model

Every indexed code entity is stored as:

```python
@dataclass(frozen=True)
class CodeEntity:
    name: str              # "login", "UserService", "handleAuth"
    file_path: str         # "src/auth/login.py"
    code: str              # Full source code of the entity
    line_start: int        # 1-indexed
    line_end: int
    language: Language      # PYTHON | JAVASCRIPT | TYPESCRIPT
    kind: EntityKind        # FUNCTION | METHOD | CLASS | INTERFACE | TYPE | ARROW_FUNCTION
    calls: tuple[str, ...]  # Functions this entity calls
    called_by: tuple[str, ...]  # Functions that call this entity
    class_name: str | None  # Parent class (for methods)
    exported: bool          # True for exported functions/classes
    embedding: list[float]  # 384-dimensional vector
    id: str                 # SHA-256 of "file_path:name:line_start"
```

## Incremental indexing

Clean doesn't re-index everything on every update. It uses two change detection strategies:

1. **Git-based** (preferred): Compares current `HEAD` against the stored commit SHA. Uses `git diff` to identify added, modified, and deleted files. Only re-parses and re-embeds changed files.

2. **Hash-based** (fallback): Computes SHA-256 of each file's content and compares against stored hashes. Slower but works for non-git directories.

Clean also detects **stale indexes** automatically. If the git HEAD has changed or uncommitted changes exist, it re-indexes before returning search results.

## Supported languages

| Language | Extensions | Parser |
|----------|-----------|--------|
| Python | `.py` | tree-sitter-python |
| JavaScript | `.js`, `.jsx`, `.mjs`, `.cjs` | tree-sitter-javascript |
| TypeScript | `.ts`, `.tsx` | tree-sitter-typescript |

<Note>
  **On the roadmap** — Support for Go, Rust, Java, and other languages is planned. Adding a new language requires a tree-sitter grammar and a parser module. See the [contributing guide](/resources/contributing).
</Note>
